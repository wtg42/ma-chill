## Context

ma-chill 為麻將終端遊戲，使用 Bun + OpenTUI React reconciler 建構 UI。目前 `tui/src/tiles/` 已有完整的牌面文字渲染邏輯（每張牌 7 字元寬 × 4 行高），但尚無主桌面 UI 實作。此設計為 MVP 第一階段：以假資料驗證視覺渲染，不接入遊戲邏輯。

## Goals / Non-Goals

**Goals:**
- 實作四列式垂直堆疊桌面，確認比例與空間分配可行
- 實作最小視窗守衛，阻止在過小視窗遊玩
- 實作玩家列固定資訊欄（右側）
- 所有元件以假資料（hardcode）驅動，純視覺驗證

**Non-Goals:**
- 接入真實遊戲邏輯或狀態管理
- 棄牌 dialog 實作（後續 change）
- AI 決策或任何後端邏輯

## Decisions

### 決策一：Layout 採方案 A（垂直堆疊）

採用單一 `flexDirection="column"` 根容器，由上至下：北家、西家、東家（各 `flexGrow={1}`）、南家（玩家，`height={20}`）。

**捨棄方案 B（模擬真實桌面方位）的原因**：Layout 複雜度高，MVP 階段不值得投入；垂直堆疊可先驗證牌面尺寸與空間，之後再重構 layout。

### 決策二：最小視窗守衛於 root 層判斷

在 App root 使用 `useTerminalDimensions()` 做尺寸判斷。`width < 140 || height < 40` 時整個畫面替換為 `<TooSmallWarning>`，尺寸恢復後自動回到遊戲畫面。

**選擇 root 層的原因**：簡單可靠，避免子元件在過小視窗渲染造成 layout 錯亂。

### 決策三：玩家資訊欄常駐固定

玩家列右側資訊欄（約 12 字元寬）固定佔位，不動態出現/消失。摸牌時顯示摸到的牌，棄牌後顯示棄出的牌，平時顯示空白或提示文字。

**選擇常駐的原因**：避免 layout 跳動；寬度計算穩定（16 張牌 × 7 = 112 字元 + 資訊欄 12 + padding，合計約 130，在 MIN_WIDTH=140 內安全）。

### 決策四：AI 列高度由 flexGrow 分配

三個 AI 列各設 `flexGrow={1}`，平均分配玩家列以外的剩餘空間。不設 minHeight，假資料階段以目視確認 AI 區高度是否足夠顯示資訊。

## Risks / Trade-offs

- **字元寬度不一致** → 含中文字（萬/東/南/西/北）或 Unicode 符號時，不同 terminal 字體可能造成對齊偏移。緩解：驗證期間在多個 terminal 測試。
- **牌面 4 行高限制 AI 列** → 若 AI 列高度不足 4 行則牌面無法完整顯示。緩解：最小高度 40 行，三個 AI 列各至少約 6 行（(40-20)/3），足夠顯示一行牌背 + 一行棄牌 + 資訊行。
- **假資料不反映真實狀態複雜度** → 後續接入真實邏輯時可能需要調整 layout。接受此風險，MVP 目的即為視覺驗證。
